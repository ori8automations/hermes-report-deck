# Hermes Report Deck

Hermes Dashboard plugin for browsing Markdown reports generated by agents, cron jobs, and automation runs, with safe folder visibility controls.

> **Early / community package.** Report Deck is functional and tested, but the Hermes dashboard plugin API surface may change before a 1.0 release. Pin your version if you need stability.

## Why this exists

Hermes agents are good at producing output: daily briefings, deployment summaries, research digests, crawl results, scan notes, handoffs, and receipts. The problem is that those reports tend to land as loose Markdown files that are technically saved but practically invisible.

**Reports need a shelf.** Report Deck gives generated reports a simple dashboard home:

```text
report files on disk → Report Deck tab → filter + read
```

**Automation readouts should keep report content read-only.** Report Deck lists and renders local Markdown reports only. It does not edit, delete, post, trigger agents, or call external services. The only write is a small local folder-visibility preference saved outside the report root.

**Frontmatter should be useful, not mandatory.** Reports with YAML frontmatter get title, date, source, lane, tags, summary, and related metadata. Plain Markdown files still show up, with the title inferred from the first heading or filename. If a report root already has an `index.json`, Report Deck uses that metadata and stable ids first.

The scope is deliberately narrow: browse reports, filter reports, read reports. That is enough to turn “the agent wrote a file somewhere” into something you can actually use from Mission Control.

## Install

Copy the plugin directory into your Hermes plugins path, enable it, and restart the dashboard:

```bash
cp -r hermes-report-deck ~/.hermes/plugins/
hermes plugins enable hermes-report-deck
hermes dashboard --host 127.0.0.1 --port 9119 --no-open
```

Manual config fallback:

```yaml
plugins:
  enabled:
    - hermes-report-deck
```

Then restart the dashboard, or rescan if it is already running:

```bash
curl http://127.0.0.1:9119/api/dashboard/plugins/rescan
```

The **Report Deck** tab appears in the dashboard navigation.

## Report root setup

Report Deck reads Markdown files from one configured report root.

| Env var | Default | Purpose |
|---|---|---|
| `REPORT_DECK_ROOT` | `$HERMES_HOME/reports` | Directory scanned recursively for `.md` and `.markdown` reports |
| `REPORT_DECK_CONFIG` | `$HERMES_HOME/report-deck.json` | Local JSON file for folder-visibility preferences |

Try it with the included samples:

```bash
REPORT_DECK_ROOT="$(pwd)/sample-reports" hermes dashboard --host 127.0.0.1 --port 9119 --no-open
```

The plugin resolves every discovered file and skips anything outside the configured root, including symlink escapes.

## Report format

A report is any `.md` or `.markdown` file under `REPORT_DECK_ROOT`. If the root contains an `index.json` with a top-level `reports` list, Report Deck uses that index for stable ids and metadata, and reads each report from its `content_path`. Without an index, it discovers Markdown files recursively and reads optional YAML frontmatter. The folder panel can hide top-level folders from listing/facets/detail access without moving or editing report files:

```markdown
---
id: nightly-crawl-2026-06-01
title: Nightly crawl summary
generated_at: 2026-06-01T02:14:00Z
source: scheduler
lane: automation
tags: [crawl, nightly]
summary: One-line summary shown in the list.
related:
  - deploy-summary-2026-06-18
---

# Nightly crawl summary

Body Markdown goes here…
```

Supported metadata:

| Field | Purpose |
|---|---|
| `id` | Optional stable report id; otherwise derived from filename |
| `title` | Display title; otherwise first heading or filename |
| `generated_at` / `date` | Sort and date filter value |
| `source` | Filterable source such as `scheduler`, `agent`, or `research` |
| `lane` | Filterable lane or project bucket |
| `tags` | Filterable tags |
| `summary` | Short list-view summary |
| `related` | Optional related report ids |

Supported inline Markdown: headings, lists, `code`, fenced code blocks, `**bold**`, `*italic*`, and links. Raw HTML is not rendered.

## Features

| Area | What it does |
|---|---|
| Report list | Shows indexed reports, or every Markdown report under the configured root if no index exists |
| Metadata | Parses optional YAML frontmatter with PyYAML or a small fallback parser |
| Filters | Filter by lane, source, tag, and date prefix |
| Reader | Opens a report body with safe lightweight Markdown rendering |
| Facets | Shows discovered lanes, sources, and tags |
| Plain Markdown | Files without frontmatter still appear |
| Samples | Includes fake sample reports for local testing |
| Folder visibility | Optionally show all folders or only selected top-level folders |

## API

Mounted by Hermes at:

```text
/api/plugins/hermes-report-deck
```

| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | Report root, existence, and report count |
| GET | `/reports` | List report metadata and facets; supports `lane`, `source`, `tag`, `date` filters |
| GET | `/reports/{id}` | Read one report's metadata and Markdown body |
| GET | `/folders` | List top-level folders under the root and current visibility state |
| PUT | `/folders` | Save visible folder preferences (`mode`: `all` or `selected`) |

`date` accepts a prefix: `YYYY`, `YYYY-MM`, or `YYYY-MM-DD`.

## Folder visibility

By default every top-level folder under the report root is visible. Use the **⚙ Folders** panel to switch to **Selected only** and choose which folders appear. Hidden folders are excluded from listing, facets, counts, and direct report reads. Folder names are single path segments only; `/`, `\`, and `..` are rejected.

## ⚠️ Safety posture

Report Deck keeps report content read-only.

- No route creates, updates, deletes, edits, imports, exports, posts, or dispatches report content.
- The only non-GET route is `PUT /folders`, which writes local folder-visibility preferences to `REPORT_DECK_CONFIG` outside the report root.
- The backend only reads Markdown files it has discovered under `REPORT_DECK_ROOT`.
- A request id is matched against discovered report ids; it is never converted directly into a filesystem path.
- Traversal-looking ids are rejected.
- Symlinks that resolve outside the report root are skipped.
- Markdown renders to React text nodes; raw HTML and active content are not rendered.
- Links are restricted to `http`, `https`, and `mailto`.
- The frontend only calls its own same-origin plugin API.

## Known limitations

- This is a report browser, not a report editor.
- There is no full-text search yet; filtering is metadata-based.
- Markdown rendering is intentionally lightweight, not a full GitHub-flavored Markdown renderer.
- Large report directories are scanned from disk on request; this is fine for a local dashboard shelf, not a search index.
- There is no built-in authentication layer beyond your Hermes dashboard deployment.

## Tests

```bash
./tests/run_tests.sh
```

The runner auto-selects a Python interpreter that has the dashboard test dependencies. It checks `$PYTHON`, an active `$VIRTUAL_ENV`, `/opt/hermes/.venv/bin/python`, then `python3` / `python`.

To force the Hermes venv:

```bash
PYTHON=/opt/hermes/.venv/bin/python ./tests/run_tests.sh
```

The suite runs `py_compile`, `node --check` if available, and a FastAPI `TestClient` smoke test covering health, list, read, filters, missing-report `404`, path-traversal rejection, outside-root safety, and fallback frontmatter parsing.
