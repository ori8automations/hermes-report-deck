# Hermes Report Deck

A **read-only** dashboard plugin for the
[Hermes Agent](https://hermes-agent.nousresearch.com) that browses Markdown
reports generated by agents or automation runs.

Agents and cron jobs are good at *producing* reports — nightly crawls, deploy
summaries, research digests, scan results — but those tend to pile up as loose
Markdown files that nobody reads. Report Deck gives you a simple in-dashboard
place to **list, filter, and read** them, with optional metadata for
organization. It never writes, edits, deletes, or calls out to the network.

```
report files on disk  →  Report Deck tab  →  filter + read
```

## Features

- **Report Deck tab** in the Hermes Dashboard.
- **Lists** every `.md` / `.markdown` file under a configured report root.
- **Optional YAML frontmatter** for title, date, source, lane, tags, summary.
  Files without frontmatter still show up (title from the first heading).
- **Filter** by lane, source, tag, and date; click a tag chip to filter.
- **Reads** a report as safe Markdown — rendered into React text nodes only, so
  there is **no raw HTML and no active content**; links are limited to
  `http(s)`/`mailto`.
- **Dark-mode first**, with a light fallback.
- **Read-only and local**: no write/delete/edit routes, no external calls.

## Requirements

- A running Hermes Agent dashboard (FastAPI backend).
- No build step, no npm dependencies (the frontend uses the Hermes Plugin SDK).
- Optional: `PyYAML` for frontmatter parsing. If it isn't installed, a small
  built-in fallback parser handles the common `key: value` / list cases.

## Install

Copy the plugin into your Hermes plugins path, enable it, and (re)start the
dashboard:

```bash
cp -r hermes-report-deck ~/.hermes/plugins/
hermes plugins enable hermes-report-deck
hermes dashboard --host 127.0.0.1 --port 9119 --no-open
```

> Copy the repository directory itself (the one containing `plugin.yaml` and
> `dashboard/`) to `~/.hermes/plugins/hermes-report-deck`. The extra
> `sample-reports/` and `tests/` folders are harmless — Hermes only reads
> `plugin.yaml` and `dashboard/`.

**Manual config fallback** — if you manage plugins via config instead of the
CLI, add it to your Hermes `config.yaml`:

```yaml
plugins:
  enabled:
    - hermes-report-deck
```

then restart the dashboard (or force a rescan if it's already running):

```bash
curl http://127.0.0.1:9119/api/dashboard/plugins/rescan
```

The **Report Deck** tab then appears in the dashboard nav.

## Configuration

| Env var | Default | Purpose |
|---|---|---|
| `REPORT_DECK_ROOT` | `$HERMES_HOME/reports` | Directory scanned for report files |

The plugin reads Markdown recursively under this single root and never reads
outside it. Point it at the bundled samples to try it out:

```bash
REPORT_DECK_ROOT="$(pwd)/sample-reports" hermes dashboard --port 9119 --no-open
```

## Report file format

A report is any `.md` / `.markdown` file under the report root. Frontmatter is
**optional** — include a YAML block at the top to enable metadata and filters:

```markdown
---
id: nightly-crawl-2026-06-01      # optional; derived from the filename if omitted
title: Nightly crawl summary       # optional; falls back to the first heading
generated_at: 2026-06-01T02:14:00Z # or `date:`  — used for sorting + the date filter
source: scheduler                  # optional; filterable
lane: automation                   # optional; filterable
tags: [crawl, nightly]             # optional; filterable
summary: One-line summary shown in the list.
related:
  - deploy-summary-2026-06-18
---

# Nightly crawl summary

Body Markdown goes here…
```

Supported inline Markdown: headings (`#`–`###`), lists, `code`, fenced code
blocks, `**bold**`, `*italic*`, and links. Raw HTML is not rendered.

## Sample reports

`sample-reports/` contains four fake reports (three with frontmatter, one
without) so you can see listing, filtering, and reading immediately — see the
`REPORT_DECK_ROOT` example above.

## Safety posture

- **Read-only:** only `GET` routes exist. No create/update/delete/edit.
- **Confined to the report root:** the backend only reads Markdown files it has
  discovered under `REPORT_DECK_ROOT`; every path is resolved and verified to
  stay within the root (symlinks that escape are skipped).
- **No path from user input:** a report is looked up by matching a discovered
  id, so a request id is never turned into a filesystem path; traversal-looking
  ids are also rejected outright.
- **No raw HTML / active content:** Markdown renders to React text nodes; links
  are restricted to `http(s)`/`mailto`.
- **No external network calls** and **no agent tools**.

## API routes (mounted at `/api/plugins/hermes-report-deck`)

| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | Report root, existence, and report count |
| GET | `/reports` | List metadata + facets (filters: `lane`, `source`, `tag`, `date`) |
| GET | `/reports/{id}` | One report's metadata + Markdown body |

`date` accepts a prefix: `YYYY`, `YYYY-MM`, or `YYYY-MM-DD`.

## Tests

```bash
./tests/run_tests.sh
```

Runs `py_compile`, `node --check` (if node is present), and a FastAPI
`TestClient` smoke suite against the bundled `sample-reports/` — covering the
health/list/read routes, each filter, a missing-report `404`, path-traversal
rejection, and a check that no file outside the report root is ever read.

The runner auto-selects a Python that has the test deps: it prefers `$PYTHON`,
then an active `$VIRTUAL_ENV`, then `/opt/hermes/.venv/bin/python`, then
`python3`. The system `python3` usually lacks `fastapi`, so if you installed
the dashboard in a venv just let it pick that up, or point at it explicitly:

```bash
PYTHON=/opt/hermes/.venv/bin/python ./tests/run_tests.sh
```

(Needs `fastapi` + `httpx` in the chosen interpreter; `PyYAML` optional.)
